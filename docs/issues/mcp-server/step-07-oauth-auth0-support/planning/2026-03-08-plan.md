# Step 07 plan: Auth0/OAuth support for Lifeline MCP

## Objective

Add Auth0/OAuth support to the public Lifeline MCP service while keeping the existing API-key path fully functional.

## Planned architecture

### Public edge

- keep `POST /mcp` as the single MCP entrypoint
- keep API-key auth support through bearer or `x-api-key`
- add Auth0 bearer-token support when MCP Auth0 config is present
- expose:
  - `GET /.well-known/oauth-protected-resource/mcp`
  - `GET /.well-known/oauth-authorization-server`

### Auth selection

1. prefer `x-api-key` when present
2. if bearer token looks like a JWT and OAuth is enabled, use Auth0 token verification
3. otherwise treat the bearer token as an MCP API key

### Principal convergence

- normalize both auth methods into the same Lifeline MCP principal contract
- preserve current scope checks in the tool layer
- avoid tool-layer branching by auth type

### Backend addition

Add one bounded internal endpoint for validated OAuth claims to become a normalized MCP principal:

- `POST /internal/mcp/auth/resolve-oauth-principal`

This use-case should rely on the existing Auth0-backed user repository behavior rather than duplicate persistence logic in the MCP service.

## Implementation workstreams

1. extend MCP config/env parsing for Auth0 issuer, audience, scopes, and metadata
2. add Auth0 JWT verification using JWKS
3. add request-level dual-auth orchestration
4. extend backend internal adapter for OAuth principal resolution
5. add integration and route tests for happy path and negative cases
6. update canonical API, backend, operations, architecture, and ADR docs
7. retain scoped issue-history artifacts for discovery, plan, implementation, and final closeout

## Validation plan

- MCP service tests for OAuth metadata, valid bearer-token flow, expired token rejection, wrong-audience rejection
- backend internal MCP auth tests for OAuth principal resolution and missing-subject rejection
- editor error scan on changed files
- deployment readiness review against current deploy-branch production flow

## Deployment plan

Use the existing production path only:

1. merge to `main`
2. promote to `deploy`
3. allow the deploy-branch workflow to package, release, restart, and verify
4. run post-deploy checks for health, OAuth metadata, API-key regression, and bearer-token validation
