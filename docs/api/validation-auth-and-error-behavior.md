# Validation, Auth, and Error Behavior

## Purpose

This document describes the cross-cutting API behavior for authentication enforcement, request validation, OAuth-related errors, and disabled notification endpoints.

## Auth enforcement model

### Global `/api` chain

Most `/api` requests pass through:

- JWT validation via `checkJwt`
- current-user attachment via `attachCurrentUser`

### Route-level guards

Protected routes then add helpers such as:

- `requireAuth()`
- `requireRole()`
- `requirePaid()`

### Canonical unauthenticated response pattern

When `requireAuth()` blocks a request, the API returns a `401` error conveying that the feature requires login and that guest mode remains local-only.

## Todo validation behavior

### Create validation

`validateTodoCreate` currently validates fields such as:

- title required and length-bounded
- optional description
- due date as `YYYY-MM-DD`, ISO date, empty string, or null
- recurrence as object, string, or null
- tags as structured objects
- `isFlagged`
- duration between 0 and 1440 minutes
- priority limited to `high`, `medium`, or `low`
- optional due time and subtasks

### Update validation

`validateTodoUpdate` uses a similar but optional-field update schema.

### Batch validation

`validateTodoBatch` requires:

- `action` in `delete`, `complete`, or `uncomplete`
- `ids` as a non-empty array of UUIDs

## Error response shape

The global error handler normalizes errors into a shared JSON shape:

- `status`
- `message`
- optional `stack` in development

## OAuth error mapping

The error handler maps common auth-library failures into friendlier `AppError` responses.

Current mappings include:

- missing or invalid authorization header -> `401`
- invalid or expired access token -> `401`

## Route-level non-AppError behavior

When route handlers throw plain errors, the global error handler wraps them into `AppError` using:

- `error.statusCode || 500`
- `error.message || 'Internal Server Error'`

This creates a more uniform response surface even when handlers mix direct `res.status(...).json(...)` branches with `next(err)`-style errors.

## Validation-vs-handler inconsistency note

The current codebase still mixes two patterns:

- centralized `next(err)` error flow
- direct `res.status(...).json({ error: ... })` responses in some handlers

Canonical API docs should therefore describe the dominant error behaviors per route group, while acknowledging that not every route currently emits the exact same JSON envelope.

## Notifications-disabled current state

Notifications are intentionally disabled in the current runtime.

Current endpoint behavior:

- `GET /api/notifications/pending` returns an empty array
- `POST /api/notifications/schedule` returns `410`
- `PATCH /api/notifications/:id/sent` returns `410`
- `DELETE /api/notifications/:id` returns `410`

The `410` responses come from `NotificationService.getDisabledResponse()` and indicate that notifications are not supported in the PostgreSQL-only runtime.

## Swagger contract generation note

The served Swagger spec is generated by merging:

- the base static Swagger file
- JSDoc-generated route documentation

Because of that, route code and validators remain the primary source of truth when there is any discrepancy between static documentation and runtime behavior.
